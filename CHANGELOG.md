### PRD-030 follow-up — Optional mobile field on contact form — 2026-05-01

Adds an optional Mobile (phone) field to the public landing-page contact form between Email and Message. Free-text input (international formats vary too widely for a useful regex), 30-character cap enforced at both Zod and DB. Threaded end-to-end: schema → form → route → service → DB column → email body. Empty submissions normalise to `NULL` in the DB and are omitted from the operator's notification email so the email stays clean for submitters who skip the field.

- **Schema** (`lib/schemas/contact-schema.ts`): new `mobile: z.string().trim().max(30).optional()` between `email` and `message`. Empty strings pass max(30) and the route normalises them to `null`.
- **Migration** (`docs/030-landing-page-polish/002-add-mobile-to-contact-submissions.sql`): `ALTER TABLE contact_submissions ADD COLUMN mobile TEXT CHECK (mobile IS NULL OR char_length(mobile) <= 30)`. Nullable column — no backfill needed for existing rows. **User runs against dev DB before merge.**
- **Form** (`app/_components/landing-contact-form.tsx`): new field with `type="tel"`, `autoComplete="tel"`, "+1 555 123 4567" placeholder, and an "(optional)" muted-text suffix on the label so users know they can skip it.
- **Route** (`app/api/contact/route.ts`): normalises the trimmed mobile value to `null` when empty, then threads it through to both `persistSubmission` and `notifyOperator`.
- **Service** (`lib/services/contact-service.ts`): `PersistInput` and `NotifyInput` gain `mobile: string | null`. Persist writes the value (or `NULL`) into the new column. The notification email's `buildNotificationHtml` conditionally inserts a `<strong>Mobile:</strong> ...` line only when present — when the submitter skipped the field, the email shows just the from-line and the message with no empty mobile row.
- **Logging** updated to include mobile presence at the redacted info level (`mobile: present | absent`) — the value itself is never logged at info/error levels per the contact service's existing redaction posture.
- **No new tokens, no new dependencies.** The Mobile field uses the same shadcn `Input` primitive and existing label tokens as the other fields.

### PRD-030 Part 3 — Landing Page Polish: Contact Section with Working Form — 2026-05-01

Adds a Contact section above the footer of the public landing page with a real working form. The form is a 3-field react-hook-form + zod composition (Name, Email, Message) that POSTs to a new public `/api/contact` route. The route validates input with Zod, enforces a per-IP rate limit, silently absorbs honeypot submissions, persists each successful submission to a new `contact_submissions` table (RLS denies all client reads), and triggers an email notification to the operator via the existing Resend/Brevo email service with reply-to set to the submitter so the operator can reply directly from their inbox. On success the form is replaced inline by a confirmation panel; on failure a toast appears and the form re-enables for retry. The fixed nav gains a "Contact" anchor link that smoothly scrolls to the section. This is the only Part of PRD-030 that introduces a backend surface — every other Part is presentational.

- **PRD-030 P3.R1–R2 — Section placement and layout.** New `<LandingContactSection />` renders between the existing bottom CTA and the footer. Two-column desktop layout (`lg:grid-cols-2`): pitch + alternative-contact panel on the left, form on the right. Single-column on mobile. The section is full-viewport (`min-h-screen`) and uses `--surface-raised` to match the bottom CTA's visual register.
- **PRD-030 P3.R3–R5 — Form fields and behaviour.** Three fields: Name (1–100 chars), Email (valid + max 254), Message (1–2000). Validation via shared `lib/schemas/contact-schema.ts` consumed by both the form and the route — the schema is the single source of truth, no validation can drift between client and server. Client-side validation runs on submit with field-level inline error text. Submit button switches label to "Sending…" with a `Loader2` spinner while in flight. On success, the form unmounts and an inline confirmation panel ("Thanks — we'll get back to you within one business day") with a "Send another" link replaces it. On failure, a `sonner` toast surfaces the message and the form re-enables.
- **PRD-030 P3.R6 — Persistence.** New `contact_submissions` table with `name`, `email` (citext for case-insensitive future-dedup), `message`, `user_agent`, `ip_address` (inet), `created_at`. CHECK constraints enforce length caps at the DB layer in addition to Zod (defence-in-depth). RLS enabled with a single `SELECT USING (false)` policy that denies all anon and authenticated reads; no INSERT/UPDATE/DELETE policies — service-role client only. Single index on `(created_at DESC)` for the operator's eventual admin UI (deferred to backlog). Migration file: `docs/030-landing-page-polish/001-contact-submissions.sql`.
- **PRD-030 P3.R7 — Email notification.** `notifyOperator()` in `lib/services/contact-service.ts` composes a plain HTML email and calls the existing `sendEmail()` with `replyTo: input.email`. Reply-to wires the operator's inbox reply directly to the submitter — no copy-paste from the body required. The email service's `EmailPayload` interface gains an optional `replyTo?: string` field; both Resend and Brevo adapters thread it through (Resend: top-level `replyTo` field; Brevo: `replyTo: { email }`). Email send failures are logged at error level but do **not** fail the API response — the submission is durably persisted and the operator can recover from the database. `notifyOperator()` therefore returns `{ status: "sent" | "failed" }` instead of throwing.
- **PRD-030 P3.R8 — Public unauthenticated route.** `app/api/contact/route.ts` accepts unauthenticated POSTs. `middleware.ts` adds `pathname === "/api/contact"` to the public-route allowlist so the auth-redirect skip applies. Standard HTTP status codes per CLAUDE.md: 200 success, 400 validation, 413 payload-too-large, 429 rate-limited, 500 server error.
- **PRD-030 P3.R9 — Abuse protection.** Per-IP rate limit of 5 submissions per 10-minute window via in-memory `Map<ip, { count, windowStart }>` keyed by `x-forwarded-for` (Vercel-aware) with `x-real-ip` fallback. Process-local — on a serverless platform, instances may have independent counters; acceptable at this scale, CAPTCHA upgrade is on backlog. 10 KB content-length cap rejects oversized payloads before parsing. Off-screen honeypot input (`absolute left-[-9999px] size-0 opacity-0 tabIndex={-1} aria-hidden`) — humans never see it; bots that fill every field hit it and the route returns 200 silently without persisting or notifying.
- **PRD-030 P3.R10 — Logging.** Per CLAUDE.md, the route logs entry (with redacted input — name length, email domain only, message length), exit (persisted id + email status), and errors with stack traces. The submitter's full email is never logged at info/error levels. The persist and notify steps log independently so a notification failure is visible in incident triage without obscuring the successful persistence.
- **PRD-030 P3.R11 — Design consistency.** Form uses existing shadcn primitives (`Input`, `Textarea`, `Label`, `Button`). No new form components introduced. Only design tokens consumed: existing surface, border, brand-primary, status-error, text-primary/secondary.
- **PRD-030 P3.R12 — Nav anchor.** The fixed nav bar's right-side group gains a "Contact" anchor (`<a href="#contact">`) before the existing "Get Started" CTA. Smooth scroll behaviour is native via the browser's default. Layout: `flex items-center gap-4 sm:gap-6` so the anchor and CTA breathe on mobile.
- **`FormState` discriminated union** (`'idle' | 'submitting' | 'success' | 'error'`) per CLAUDE.md TypeScript Patterns — no booleans for multi-state flows.
- **New env vars:** `CONTACT_OPERATOR_EMAIL` (server-only, recipient of notification emails) and `NEXT_PUBLIC_CALENDLY_URL` (optional, public — when set, the contact section renders a "Book time directly" link).
- **No new npm dependencies.** Reuses `react-hook-form`, `@hookform/resolvers/zod`, `zod`, `sonner`, and the existing email service.

### PRD-029 Part 7 — Workspace Notifications: Per-User Fan-Out & Actor Suppression — 2026-05-01

Closes a real read-isolation bug and a personal-workspace gap by restructuring storage from one-row-per-event to **one-row-per-recipient**. Broadcasts are fanned out at emit time into per-recipient rows; each user has their own row with their own `read_at`. Personal-workspace notifications work natively. Actor suppression is built in. Schema migration included; in-flight broadcast rows are dropped at cutover (PRD §P7.R5). The bell, dropdown, hooks, renderers, and Realtime subscription are unchanged — they were already per-row, which is now per-user by construction.

- **PRD-029 P7.R1 — Read state is per-user.** Marking a notification read by user A no longer affects user B's view. Each recipient has a separate row with its own `read_at`; cross-user read leakage is structurally impossible. The bell's existing read/unread logic (`mark-read`, `mark-all-read`, `unreadCount`, the session-scoped filter from Part 6) all continue to work — they key by row, and per-row is now per-user.
- **PRD-029 P7.R2 — Personal workspaces receive notifications.** `workspace_notifications.team_id` is now nullable. A targeted emit to a personal-workspace owner writes one row keyed on the owner's user_id with `team_id IS NULL`. The bell on a personal-only account is no longer permanently empty (subject to actor suppression — see P7.R3).
- **PRD-029 P7.R3 — Actor suppression.** `EmitInput` gains a top-level optional `actorId` field. When set on a broadcast emit, the actor is removed from the recipient set — user X merging a theme no longer sees a `theme.merged` row about their own action; their teammates do. Targeted emits ignore `actorId` because the named user is the audience by design (e.g., "your bulk re-extract finished" addressed to the actor stays addressed to the actor).
- **PRD-029 P7.R4 — Schema reshape.** Migration `003-fan-out-and-actor-suppression.sql` flips `team_id` to NULLABLE, `user_id` to NOT NULL, drops in-flight broadcast rows (`DELETE WHERE user_id IS NULL`), drops the broadcast-aware RLS policies and replaces them with a single `user_id = auth.uid()` predicate for both SELECT and UPDATE, and rebuilds indexes — `(user_id, created_at DESC)` and `(user_id) WHERE read_at IS NULL` lose their now-redundant partial WHEREs (since `user_id` is non-null on every row); the `team_id`-based partial indexes are dropped entirely (no consumer left).
- **PRD-029 P7.R5 — In-flight broadcasts dropped, not fanned out.** A correct fan-out migration would need defensive handling for archived/deleted teams, ambiguous recipient resolution (current vs historical membership), and a runbook. Notifications are ephemeral by design (Part 5 retention is 30/90 days when it ships); a one-time cutover loss is consistent with the surface's design and saves real complexity.
- **PRD-029 P7.R6 — Emitter call sites stay minimal.** The single existing emit call (`app/api/themes/candidates/[id]/merge/route.ts`) gains one new field (`actorId: auth.user.id`) and drops the `if (result.teamId !== null)` guard — the service handles personal workspaces natively. Future emitters (PRD-028 supersession, PRD-017 bulk re-extract) follow the same shape.
- **Architectural choice — fan-out at emit time, not a side table** (TRD §"Why Fan-Out (Not a Side Table)"). Three storage shapes were considered: per-recipient row (chosen), side table `notification_reads(notification_id, user_id, read_at)`, and per-row JSONB `read_by[]`. Fan-out wins because read paths (`markRead`, `unreadCount`, the `read_at IS NULL` partial index, Part 6's session filter) are unchanged — per-row is per-user automatically. RLS simplifies to one predicate. Realtime per-user filtering already works via RLS. Write amplification (a team of 20 × a handful of merges per day = dozens of rows) is lost in the noise at workspace scale. The TRD's original Decision 1 rejected fan-out for "no consumer needs sub-group fan-out" — that rationale doesn't apply to per-user read isolation, which IS a real consumer requirement.
- **Architectural choice — drop existing broadcast rows, don't migrate them** (TRD §"Why Drop Existing Broadcast Rows"). Reconstructing recipient sets from `team_members` at migration time is brittle when membership has changed since emit. Drop is a single `DELETE`, fits the ephemeral-by-design surface, and the alternative is a one-shot script with archived-team handling and a runbook for a tiny gain.
- **Architectural choice — explicit top-level `actorId`, not payload-derived** (TRD §"Why Explicit `actorId`, Not Payload-Derived"). `theme.merged`'s payload already carries `actorId`, so the service could read it from there. Rejected because (a) the registry would stop being a closed-box and start being introspected for special fields, (b) `payload` belongs to the renderer (Part 3) — moving routing logic into payload conflates "what's shown" with "who sees it", and (c) some emitters have no actor (system actions, scheduled jobs); top-level optionality makes that absence explicit. The cost is one new line per emit call site.
- **Architectural choice — `emit` returns `WorkspaceNotification[]`** instead of a single row. Non-breaking for the only current call site (which discards the return). The array shape is more accurate for a fan-out call and keeps the contract honest.
- **New `InvalidEmitTargetError`.** Prevents a silent no-op when neither `userId` nor `teamId` is supplied — that's a programmer error, not a "no recipients" case (which still returns `[]` cleanly when actor suppression empties the recipient set).
- **Bell `BELL_COLUMNS` join relaxed from `teams!inner(name)` to `teams(name)`.** Personal-workspace rows have no team to join; the mapper falls back to an empty `teamName`. The dropdown's per-row label suppression (Part 2) already handles the visual treatment when every visible row shares the same workspace, so personal-only bells show no label noise.
- **Increment 7.1 — Migration.** `docs/029-workspace-notifications/003-fan-out-and-actor-suppression.sql` written. Drop existing broadcasts, alter columns, swap policies, rebuild indexes. User applies against dev DB.
- **Increment 7.2 — Service + emitter wiring.** Repository (`notification-repository.ts`, `supabase-notification-repository.ts`) gains `bulkInsert` + `listActiveTeamMemberIds`; `BELL_COLUMNS` relaxed; `toBellNotification` falls back to `"Personal"` (audit fix B). Service (`notification-service.ts`) gains `actorId` on `EmitInput`, `resolveRecipients` helper, bulk-insert flow, `InvalidEmitTargetError`. Merge route adds `actorId: auth.user.id`, drops the `teamId !== null` guard. Type (`lib/types/notification.ts`) updated for new column nullability. **`npx tsc --noEmit` passes clean.**
- **Increment 7.3 — End-of-part audit.** Three real fixes produced:
  - **Fix A — Dead code removal: `repo.insert` (singular).** Pre-Part-7, `insert` was the single-row write path. Part 7's `emit` exclusively uses `bulkInsert` (it always operates on a recipient array — even targeted emits resolve to a one-element array). A grep confirmed zero callers of `insert`. Removed from the `NotificationRepository` interface and the Supabase implementation. Per CLAUDE.md "Delete dead code immediately." The `bulkInsert` early-return on empty input handles the degenerate one-row case without a separate code path.
  - **Fix B — UI regression: empty `teamName` for personal-workspace rows.** With `team_id` nullable, a user holding both team and personal notifications would have hit a per-row workspace label rendering as `<bullet> <empty span>` in the dropdown's cross-workspace mode. The mapper's `?? ""` fallback was the cause. Changed to `?? "Personal"` so the workspace label reads cleanly. The dropdown's existing label suppression (renders no label when every visible row shares one workspace) still keeps personal-only bells label-free.
  - **Fix C — Misleading log message in `emit`'s zero-recipient branch.** The log claimed "after actor suppression" but the same branch fires for an empty team (no active members) and for an actor-only personal workspace. Reworded to enumerate all three causes so logs are accurate during incident triage.
- **Net code change.** New: `docs/029-workspace-notifications/003-fan-out-and-actor-suppression.sql`. Modified: 6 source files. ~80 lines added, ~10 removed (audit Fix A removed ~25 lines of `insert` implementation + interface entry). No new dependencies, no new env vars, no UI component changes (bell, dropdown, hooks, renderers untouched). **Final `npx tsc --noEmit` passes clean** after every audit fix.

### PRD-029 Part 6 — Workspace Notifications: Session-Scoped Read Filter & Expiry Filter — 2026-05-01

Two narrow filters added to the bell so the dropdown stops drifting from "what's actionable now" to "everything in the last 30 days." Server-side: expired rows (those with `expires_at` in the past) are no longer returned by the listing query or counted by the unread badge. Client-side: read notifications are scoped to the current app-mount session — they stay visible while the user is using the app and disappear on the next hard reload, new tab, or browser restart. Unread notifications are unaffected by both filters' read-state logic. No schema migration, no new API surface, no Realtime change.

- **PRD-029 P6.R1 — Session-scoped read filter.** `NotificationBell` captures `sessionStartedAt = Date.now()` once at mount via `useState(() => Date.now())` and threads it through `NotificationDropdown` into `useNotifications({ sessionStartedAt })`. The hook keeps its full server-returned row set in internal `allRows` state (so pagination cursors stay valid even when filtering trims the rendered list) and exposes a `useMemo`-derived `rows` filtered by `r.readAt === null || new Date(r.readAt).getTime() >= sessionStartedAt`. Hard reload remounts the bell → fresh `sessionStartedAt` → reads from prior sessions fall away on the next render. Soft Next.js client navigation does not remount the sidebar-mounted bell, so the session persists across in-app navigation.
- **PRD-029 P6.R2 — Server-side expiry filter.** New `applyNotExpiredFilter()` helper in `lib/repositories/supabase/supabase-notification-repository.ts` adds `(expires_at IS NULL OR expires_at > now())` via PostgREST `.or()`. The dropdown listing and the badge count now both stop returning rows whose emitter has explicitly marked them stale. `deleteExpired` continues to handle actual row removal when Part 5's deferred cleanup ships; this part only changes what the bell *renders*.
- **PRD-029 P6.R3 — Per-tab session boundaries.** Each tab's `sessionStartedAt` is independent. A notification marked read in Tab A renders in read style in Tab B (driven by Realtime refetch) and continues to render in Tab B until Tab B is reloaded. Per-tab read state tracking via `BroadcastChannel` / `localStorage` is explicitly out of scope — the trade is acceptable for a glance surface.
- **Architectural choice — client-side, not server-side, for the read filter** (TRD §"Why Client-Side, Not Server-Side, for the Read Filter"). Two reasons: (1) cursor pagination would break — a keyset cursor returned at T=0 may reference a row that's filtered at T=20s, producing duplicate or missing boundaries on "Load more"; and (2) `sessionStartedAt` is a UI concern, not a data-visibility rule. Server stays focused on RLS + expiry; client owns "what does this UI surface want to render right now". Future surfaces (e.g., an inbox page that wants all read notifications) get this for free.
- **Architectural choice — server-side, not client-side, for the expiry filter** (TRD §"Why Client-Side, Not Server-Side, for the Read Filter"). `expires_at` is a true visibility rule that every consumer surface should respect — emitters explicitly marking a row stale should not see that row anywhere, not just in the bell.
- **Architectural choice — no `setInterval` / tick** (TRD §"Why No setInterval / Tick"). The session-scoped filter is static for the session lifetime: `sessionStartedAt` is captured once and never changes. A row's `readAt` only changes via mark-read flows (which already trigger Realtime refetch + state update). There's no monotonic time threshold ticking past values, so no re-render cadence is needed. This is the key simplification over the originally-considered 15-second-fade design.
- **Architectural choice — session = app-mount lifetime** (TRD §"Why Session = App-Mount Lifetime"). Three scopes were considered: app-mount (in-memory, resets on hard reload / new tab / browser restart), tab-session (`sessionStorage`, survives in-tab reload — counter-intuitive against P6.R1), and auth-session (server-side or `localStorage`, requires bridging auth state changes). App-mount matches the PRD's spec literally and requires zero storage and zero auth-state subscription.
- **Forward-compat realised, not deferred.** The hook's `sessionStartedAt: number` shape extends to a `BroadcastChannel`-synced or `localStorage`-backed value if cross-tab read state ever lands as a real requirement; the contract does not change.
- **Increment 6.1 — Filters & wiring.** Single increment. Files modified: `lib/repositories/supabase/supabase-notification-repository.ts` (helper + two `applyNotExpiredFilter` call sites), `lib/hooks/use-notifications.ts` (new `UseNotificationsOptions` interface, internal `rows` → `allRows`, `useMemo`-derived filtered `rows`), `components/notifications/notification-bell.tsx` (capture `sessionStartedAt`), `components/notifications/notification-dropdown.tsx` (forward through to hook). ### PRD-029 Part 4 — Workspace Notifications: Delivery Mechanism — 2026-04-30

Moves the bell from "polling within 30 seconds" (Part 2) to "Realtime within 1-2 seconds, polling at 5-minute cadence as a safety net for silent WebSocket drops." The project's first use of Supabase Realtime — `@supabase/supabase-js` was already in tree, just unused. One SQL migration enables Realtime publication on `workspace_notifications`; one new shared hook (`useNotificationRealtime`) wraps the subscription lifecycle; the two existing fetch hooks each gain a one-line Realtime subscription call. Public hook return shapes are byte-identical to Part 2 — `NotificationBell` and `NotificationDropdown` consume them unmodified, just with sub-second latency.

- **PRD-029 P4.R1 — Notifications appear within an acceptable window of being emitted.** Realtime delivers within 1-2 seconds in the happy path; polling at 5-minute cadence is the floor when Realtime fails silently. A colleague verbally telling another user "I just merged a theme" no longer beats the badge by minutes.
- **PRD-029 P4.R2 — Hybrid mechanism documented.** Three options were considered: pure polling (current Part 2 — 30s floor violates "within seconds"), pure Realtime (subsecond happy path but indefinitely stale on silent drops), and hybrid (Realtime primary + polling backstop). Hybrid chosen because notifications are user-trust load-bearing — "stale forever" is a worse failure mode than "stale for 5 minutes." TRD §"Why Hybrid (Not Pure Realtime, Not Pure Polling)" pins the rationale.
- **PRD-029 P4.R3 — Idle tabs do not drain resources.** Polling pauses entirely when `document.visibilityState !== "visible"` (Part 2 mechanism unchanged); on focus, polling restarts and refetches immediately. Realtime subscription continues in the background regardless — the WebSocket is cheap to hold open.
- **PRD-029 P4.R4 — Reconnect & resync.** Supabase JS auto-reconnects dropped WebSocket subscriptions; the polling fallback ensures the badge corrects within 5 minutes even if the reconnect logic muffs an event. On focus after a hidden window, polling refetches immediately — correctness recovers without manual page reload.
- **PRD-029 P4.R5 — Mechanism does not block the user-perceived UI.** Subscription setup, polling ticks, and resync all happen in `useEffect` background; a slow or failing subscription never delays render or interaction.
- **Architectural choice — subscribe without an explicit filter, rely on RLS-on-Realtime.** Supabase Realtime's `filter` parameter only supports single-equality predicates; our visibility rule (`user_id = auth.uid() OR (user_id IS NULL AND team_id IN user's teams)`) is a disjunction Realtime cannot express. The chosen path: subscribe to `workspace_notifications` table-wide and let RLS-on-Realtime filter the event stream per subscriber server-side. Each user receives events only for rows their SELECT policy lets them see.
- **Architectural choice — shared channel name, per-consumer channel objects.** Both consumer hooks (`use-unread-count`, `use-notifications`) call `supabase.channel("workspace-notifications")`. Each call creates a distinct channel object (no client-side dedup with default opts), but they multiplex onto one underlying WebSocket. Per-consumer channels mean each cleanup is unambiguously local — `supabase.removeChannel(channel)` only tears down this consumer's listener.
- **Architectural choice — ref-stabilised callback in `useNotificationRealtime`.** Without ref-stabilisation, every parent re-render that produced a new `onChange` identity would tear down and re-mount the subscription — a real WebSocket round-trip cost. The ref pattern keeps subscription lifecycle tied to mount/unmount only.
- **Architectural choice — 5-minute polling fallback, not 30-second.** In Part 2 polling was the primary signal at 30s; in Part 4 polling is the backstop and bumps to 5min. Realtime is what meets the latency bar; polling exists only because Realtime can fail silently and we want a known-bounded staleness window.
- **Increment 4.1 — Migration + shared subscription hook.** Created `docs/029-workspace-notifications/002-enable-realtime-on-workspace-notifications.sql` (`ALTER PUBLICATION supabase_realtime ADD TABLE workspace_notifications`). Created `lib/hooks/use-notification-realtime.ts` — the shared hook with ref-stabilised callback, channel-name convention `"workspace-notifications"`, and `[useNotificationRealtime] subscription status: <STATUS>` logging on every state transition (`SUBSCRIBED` / `CHANNEL_ERROR` / `TIMED_OUT` / `CLOSED`). Default `REPLICA IDENTITY` (PRIMARY KEY) is sufficient — the bell uses Realtime events as a "something changed, refetch" trigger and does not read column values out of the change payload. `npx tsc --noEmit` passes.
- **Increment 4.2 — Wire badge + list hooks.** Two surgical edits:
  - `use-unread-count.ts`: bumped `POLL_INTERVAL_MS` from `30_000` to `5 * 60 * 1000`; added `useNotificationRealtime(fetchCount)` immediately after the `fetchCount` declaration; rewrote the file header comment to describe the Realtime-primary / polling-fallback design. The visibility-aware polling loop, `cancelledRef` guard, imperative `refetch` return, and `UseUnreadCountReturn` shape are unchanged.
  - `use-notifications.ts`: added `useNotificationRealtime(refresh)` after the `refresh` declaration. Subscription is bell-open-scoped (Part 2's lazy-mount via Radix Portal — the hook only mounts when the user opens the bell). Closed bell pays no Realtime cost on the dropdown side; the badge's subscription stays always-on because the trigger is always rendered.
  - Public hook return shapes byte-identical to Part 2; consumers (`NotificationBell`, `NotificationDropdown`) unchanged.
- **Increment 4.3 — End-of-part audit.** Audit produced **two real fixes** plus the standard documentation pass.
  - **Fix A — Race condition between mount-fetch and Realtime-triggered fetch.** When the bell mounts and fires the initial fetch, a Realtime event arriving moments later triggers a second fetch via `useNotificationRealtime`. Both are in flight; whichever response arrives second overwrites whichever arrived first — including the case where the Realtime-driven fetch (which contains the new notification) completes *before* the initial fetch (which doesn't), and the stale initial fetch overwrites the fresh one. Until the next Realtime event self-corrects, the badge/list shows stale state. The existing `cancelledRef` only catches unmount, not interleaved fetches. Added a `generationRef` counter incremented per call — fetches whose captured generation does not match the current one bail before mutating state. Applied to both `use-unread-count.ts` (gates `setCount`) and `use-notifications.ts` (gates `setRows` / `setCursor` / `setHasMore` and the catch-branch `setError`). `finally`'s `setIsLoading(false)` / `setIsLoadingMore(false)` stay gated only on `cancelledRef` — the loading flags can safely be flipped off by stale fetches because the latest fetch's success path already cleared them; double-flip is a no-op.
  - **Fix B — JSDoc claim about Supabase JS deduplication was incorrect.** The hook's comment said "Supabase JS deduplicates channels by name." Verified against `@supabase/realtime-js`'s `RealtimeClient.channel()` — dedup only happens when `Object.is(opts, channel.params)` matches; with default opts (object literals from each call), references differ and no dedup happens. The code's behaviour is correct (per-consumer cleanup is unambiguous because each consumer holds its own channel reference) but the rationale was wrong. Rewrote the JSDoc to accurately describe what happens: "Multiple consumers each create their own channel object under this name; supabase-js multiplexes them onto one underlying WebSocket. Because each consumer holds its own channel reference, `removeChannel` on cleanup is unambiguous."
  - **Post-ship hotfix C (2026-04-30) — Fix B's analysis was *also* wrong; the actual runtime behaviour required unique channel names per consumer.** The bell mounted in production and threw `Error: cannot add postgres_changes callbacks for realtime:workspace-notifications after subscribe()` from inside `useNotificationRealtime`'s effect. Trace: `NotificationBell` → `useUnreadCount` mounts → calls `useNotificationRealtime(fetchCount)` → opens channel `realtime:workspace-notifications` → `.subscribe()` succeeds. Then `NotificationBell`'s child `NotificationDropdown` → `useNotifications` mounts → calls `useNotificationRealtime(refresh)` → `supabase.channel("workspace-notifications")` returns the **already-subscribed** channel from the first consumer (Supabase Realtime dedupes by topic regardless of what the supabase-js source line my audit quoted suggested — either dedup is more aggressive than that branch implies, or the Realtime *server* enforces topic uniqueness on the wire and the client error surfaces it pre-flight). Adding `.on("postgres_changes", ...)` to a channel after `.subscribe()` is rejected. **Real fix:** each hook instance now gets a unique channel name via React's `useId()` — `workspace-notifications::r0:` for the badge consumer, `workspace-notifications::r1:` for the dropdown consumer. No topic collision, two server-side subscriptions multiplexed onto one WebSocket (acceptable at our scale per the TRD's "subscribe without filter" trade-off note). Status logging now includes the channel name (`[useNotificationRealtime] workspace-notifications::r0: status: SUBSCRIBED`) so production logs distinguish consumers. JSDoc rewritten to explain *why* per-consumer channel names are necessary rather than the earlier (incorrect) claim that consumers could safely share one. **Lesson learned for future audits:** the Fix B verification only re-read code; it did not exercise the runtime path with both consumers mounted simultaneously. End-of-part audits for hook code should include a "mount both consumers in the same React tree and verify subscriptions co-exist" step when the hook is shared.
  - **Verification (post-fix):** SRP confirmed (`use-notification-realtime.ts` does one thing); DRY confirmed (subscription setup in one place; both consumers call `useNotificationRealtime(callback)`); logging consistent (`[useNotificationRealtime] subscription status: <STATUS>` plus consumers' existing prefixes); zero dead code; convention compliance (`useX` naming, `LOG_PREFIX`, return shapes unchanged); zero `any` leaks (`onChange` typed `() => void` — never inspects the change payload); connection lifecycle clean (`useEffect` cleanup → `supabase.removeChannel` → sign-out cascades through `AuthenticatedLayout` swapping to a public route, unmounting the bell tree); no regressions in Part 2's behaviour or PRD-024's streaming module. `npx tsc --noEmit` passes.
- **Net code change.** New: `docs/029-workspace-notifications/002-enable-realtime-on-workspace-notifications.sql` (~15 LOC), `lib/hooks/use-notification-realtime.ts` (~70 LOC). Modified: `lib/hooks/use-unread-count.ts` (3 changes — header comment + `POLL_INTERVAL_MS` value + `useNotificationRealtime` call + `generationRef`), `lib/hooks/use-notifications.ts` (header comment + `useNotificationRealtime` call + `generationRef`). Total: ~85 LOC of new code + ~30 LOC of targeted modifications.

Table grows ~20MB/year at expected emit rates; revisit when actual size or query latency starts mattering. Per-row `expires_at` column from Part 1 stays unused until cleanup ships. ### PRD-026 Part 2 — Theme Deduplication: Platform-Suggested Merge Candidates — 2026-04-30

Surfaces a workspace-scoped, ranked list of theme pairs that look like duplicates so admins can review (and dismiss or, in Part 3, merge) without auditing the full taxonomy. Builds on Part 1's persisted theme embeddings. Cron-based background refresh deferred per scope decision; staleness handled via stale-on-mount auto-refresh on first admin visit per day.

- **PRD-026 P2.R1 — Persisted, ranked candidate list per workspace.** New `theme_merge_candidates` table (`docs/026-theme-deduplication/003-create-theme-candidates-and-dismissals.sql`) stores pairs whose embedding similarity meets `THEME_CANDIDATE_SIMILARITY_THRESHOLD` (default `0.80`, env-tunable) along with snapshot stats (assignments / distinct sessions / distinct clients / last-assigned timestamp) per side, scoring components (`similarity_score`, `volume_score`, `recency_score`, `combined_score`), shared-keywords hint, and a `refresh_batch_id`. CHECK constraint enforces normalised pair ordering (`theme_a_id < theme_b_id`) so (A, B) and (B, A) are the same row. Per-workspace unique pair indexes + per-workspace `(combined_score DESC)` indexes make page reads single-query, single-index.
- **PRD-026 P2.R2 — Composite ranking, not similarity-only.** `combined_score = 0.5·sim + 0.3·vol + 0.2·rec`. `volume_score` log-scaled around `MAX_VOLUME_NORM=200` (a pair touching 200+ signals saturates); `recency_score` exponential-decay with a 30-day half-life (`1.0` today, `~0.37` at 30 days, `~0.05` at 90 days). Worked example pinned in TRD Decision 12: a high-similarity-low-volume pair ranks below a moderate-similarity-heavy-volume pair, matching the PRD intent. Score components persisted alongside `combined_score` so the UI can show similarity directly and the admin can mentally sort by any dimension.
- **PRD-026 P2.R3 — Each candidate explains itself.** UI per row shows: both names + descriptions, confidence pill (`≥0.92` status-success / `0.85–0.92` status-warning / else neutral), per-side stats footer ("42 signals · 12 sessions · 8 clients" with proper singular/plural), a token-based shared-keywords hint, and a small score-component breakdown. Hint generation is in-process (no LLM) — case-insensitive intersection of name+description tokens minus a built-in stop-word list, capped at 3 tokens. LLM-rationale upgrade remains a backlog item.

Cost shifts from "every workspace, every day" to "first admin visit per workspace per day". ### PRD-026 Part 1 — Theme Deduplication: Embedding-Based Prevention at Theme Creation — 2026-04-30

Stops semantic-duplicate themes ("API Speed", "API Latency", "API Performance") from accumulating during signal extraction. Two layers stack: prompt bias steers the LLM toward umbrella names on first creation; an embedding similarity guard catches synonyms the LLM still proposes. Residual duplicates (rare, by design — the threshold is conservative) are absorbed by Parts 2–3 (admin-curated merge candidates and admin-confirmed merge).

- **PRD-026 P1.R1 — Themes carry vector representations.** `themes.embedding vector(1536)` column added via `docs/026-theme-deduplication/001-add-themes-embedding.sql` alongside an HNSW index (`themes_embedding_hnsw_idx`, `vector_cosine_ops`) and the forward-compat `merged_into_theme_id UUID FK` reserved for Part 3's merge transaction. Migration `002-tighten-themes-embedding-not-null.sql` lifts the column to `NOT NULL` after the backfill populates every pre-existing row. Both migrations are idempotent (`IF NOT EXISTS` on every object) so partial application replays cleanly.
- **PRD-026 P1.R2 — Semantic-similarity check before any new theme is created.** `lib/services/theme-prevention.ts` exposes `runThemePrevention()` — collects the unique proposed-new names (deduped by lowercased key, fast-path filtered against existing-theme names per P1.R6), embeds them in one batched call via `embedTexts`, computes cosine similarity in-memory against every existing theme's embedding, and returns a `Map<lowerName, PreventionDecision>`. Decisions are either `{ kind: "reuse", themeId, matchedName, score }` or `{ kind: "new", embedding, score }` — the "new" branch carries the just-computed vector so creation does not pay for a second embedding round trip. `assignSessionThemes` runs the guard as a pre-pass between the LLM call and the assignment loop; the loop consults the map per signal with no per-iteration embedding work.
- **PRD-026 P1.R3 — Conservative threshold.** `THEME_PREVENTION_SIMILARITY_THRESHOLD` env var (default `0.92`, validated at read time). Sampling on `text-embedding-3-small` for short topic labels: synonyms ("API Performance" vs "API Speed") sit at 0.93–0.96; distinct topics ("API Performance" vs "API Cost") sit at 0.78–0.84. The default sits at the gap, biasing toward false-negatives (let some duplicates through) over false-positives (collapse distinct concepts) per the PRD.
- **PRD-026 P1.R4 — Prevention is observable.** Every decision logs under `[theme-prevention] decision — sessionId: … | proposed: "X" | matched: "Y"|none | score: 0.XXX | result: reuse|new`. Per-call summary line records reused/new/total counts. The threshold is tunable from production telemetry without a code deploy.
- **PRD-026 P1.R5 — No perceptible latency.** One batched embedding call per extraction regardless of how many "new" names the LLM produced. In-memory cosine vs the existing-theme set (already loaded for the LLM prompt) — no additional DB round trips. Theme-stage timing log values (`chain timing — themes: …ms`) are unchanged in workspaces with few new proposals; in workspaces with many proposals the delta is bounded by one embedding-batch round trip (~100–300ms).
- **PRD-026 P1.R6 — Exact-name match remains the fast path.** `collectUnresolvedThemes` filters out any proposal whose lowercased name already exists in `themeByLowerName`; those proposals never reach the embedding pipeline. The assignment loop's first check is also the cache lookup, so even within a batch a second occurrence of the same proposed name short-circuits without re-running prevention math.
- **Prompt bias as a complementary first line of defense.** `lib/prompts/theme-assignment.ts` Rule 8 examples strengthened ("General Feedback", "Performance", "API" → too broad; "Button Color on Page 3", "API Speed In Reports Tab" → too narrow). New Rule 9 instructs the LLM to prefer umbrella names that fit closely-related future signals on the same topic, with worked examples in both directions and an explicit warning that the umbrella covers variations of the same topic, not different topics that share a word. Catches the cold-start case where no existing themes are around for the embedding guard to match against.
- **Single source of truth for embedding text composition.** `buildThemeEmbeddingText(name, description)` in `theme-prevention.ts` — used by both the prevention guard and the backfill script. Drift here would silently lower match quality, so both paths import the same helper.
- **Backfill script.** `scripts/backfill-theme-embeddings.ts` runs once between migrations 001 and 002. Loads env via `@next/env` (same loader Next.js uses, so the script reads exactly the same vars as `next dev` — no `dotenv` dep, no parser-mismatch surprises). Constructs a service-role Supabase client, fetches every theme with `embedding IS NULL` (via the new `themeRepo.listMissingEmbeddings()` minimal projection — id/name/description only), composes embedding text via `buildThemeEmbeddingText`, batches via `embedTexts` (the existing pipeline handles rate limits + retries), writes back via `themeRepo.updateEmbedding()`. Idempotent — re-running after a partial failure resumes cleanly. Exits non-zero if any row fails so the failure mode is visible.
- **Repository changes.** `ThemeRepository` gains `listMissingEmbeddings()` (returns `Pick<Theme, "id" | "name" | "description">[]` so backfill rows don't violate the tightened `Theme.embedding` type) and `updateEmbedding(id, embedding)` (write-only, distinct from general `update()` so the operational intent is explicit). `ThemeInsert.embedding: number[]` and `Theme.embedding: number[]` (both non-null) reflect the post-rollout DB invariant. `mapRow`'s `parseEmbedding` throws if it ever sees a NULL — defensive against a misconfigured environment, never trips post-rollout.
- **Architectural choice — two-layer defense, not just one.** Prompt-only would let synonyms leak whenever the LLM ignores the guidance (especially in cold-start workspaces). Embedding-only would pay for a round trip per synonym + risks near-misses below threshold. Both layers stack: the prompt biases first-time creation toward umbrellas (catches the cold-start case where no embedding match exists yet); the embedding guard catches synonyms the LLM still proposes despite the bias. Failures of one layer are absorbed by the other.
- **Architectural choice — in-memory cosine for prevention, RPC reserved for Part 2.** Every active workspace theme is already loaded for the LLM prompt; augmenting that read to include the `embedding` column adds one query payload, not a round trip per check. Per extraction this is O(N·D) where N is the workspace's active theme count (low thousands at most) and D=1536 — well below user-noticeable latency. Part 2's pair-generation across the full taxonomy may move to a `match_themes` RPC when workspaces grow dense; the HNSW index added in Part 1's migration is the seam.
- **Architectural choice — `merged_into_theme_id` added in Part 1, written by Part 3.** Forward-compat per the project rule that data structures should carry the shape later parts will need. Part 3's merge transaction sets the pointer + flips `is_archived`; Part 1 leaves it `NULL` everywhere. Doing both column additions in a single migration avoids a second round of `ALTER TABLE themes` against a hot table.
- **Forward-compatibility delivered, not promised.** Part 2 (admin-curated merge candidates) reads the same `embedding` column + HNSW index already in place; no additional theme-table changes. Part 3 (admin-confirmed merge) writes `merged_into_theme_id` and re-points `signal_themes.theme_id` inside a transaction; no schema change. Part 4 (notify affected users of merges) hooks into PRD-029's notification primitive via the `theme.merged` event type already registered in `lib/notifications/events.ts`.
- **What does not ship in Part 1.** No admin UI (`/settings/themes` is Parts 2–3). No merge transaction (Part 3). No notification emission (Part 4). No per-workspace tunable threshold (deferred to backlog). No LLM-judged candidate filtering (deferred to backlog).
