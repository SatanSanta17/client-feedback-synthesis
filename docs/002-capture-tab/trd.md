## Part 1: Database Schema and Client Management

### Technical Decisions

- **Database tables:** Two new tables тАФ `clients` and `sessions`. Both use UUID primary keys generated by `gen_random_uuid()`. Both include `deleted_at` for soft deletes.
- **Client name uniqueness:** Enforced at the database level with a unique index on `LOWER(name)` filtered to `WHERE deleted_at IS NULL`. This prevents "Acme" and "acme" from coexisting while allowing a soft-deleted client's name to be reused.
- **Foreign keys:** `sessions.client_id` references `clients.id`. No cascade delete тАФ deleting a client is independent of deleting sessions (both are soft deletes anyway).
- **`created_by` tracking:** `sessions.created_by` stores the Supabase Auth user ID (`auth.uid()`). This is set via RLS default or by the inserting service. It enables per-user filtering in the future without a schema change.
- **RLS policies:** Both tables are locked down to authenticated users only. The `anon` key gets no access. All policies filter `deleted_at IS NULL` for SELECT operations so soft-deleted records are invisible by default.
- **Supabase client usage:** API routes use the server-side `createClient()` from `lib/supabase/server.ts` which respects RLS via the user's session cookies. No service role key needed for these operations.
- **Client autocomplete:** A lightweight API route that queries `clients` with an `ILIKE` filter. Returns a simple array of `{ id, name }`. Debounced on the frontend (300ms) to avoid excessive calls.
- **New client creation:** Handled inline during session save, not as a separate step. When the user selects "Create [name]" from the combobox, the frontend stores the intent. On form submit, the API route creates the client first (if new), then creates the session with the resulting `client_id`. This keeps it transactional тАФ no orphan clients from abandoned forms.
- **Combobox component:** Uses shadcn/ui `Command` (cmdk) + `Popover` for the autocomplete. This is an established pattern in shadcn and handles keyboard navigation, filtering, and custom option rendering out of the box.

### Database Schema

#### `clients` table

```sql
CREATE TABLE clients (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  deleted_at TIMESTAMPTZ
);

-- Case-insensitive unique name among non-deleted clients
CREATE UNIQUE INDEX clients_name_unique ON clients (LOWER(name)) WHERE deleted_at IS NULL;

-- Enable RLS
ALTER TABLE clients ENABLE ROW LEVEL SECURITY;

-- Authenticated users can read non-deleted clients
CREATE POLICY "Authenticated users can read clients"
  ON clients FOR SELECT
  TO authenticated
  USING (deleted_at IS NULL);

-- Authenticated users can insert clients
CREATE POLICY "Authenticated users can insert clients"
  ON clients FOR INSERT
  TO authenticated
  WITH CHECK (true);

-- Authenticated users can update clients
CREATE POLICY "Authenticated users can update clients"
  ON clients FOR UPDATE
  TO authenticated
  USING (deleted_at IS NULL)
  WITH CHECK (true);
```

#### `sessions` table

```sql
CREATE TABLE sessions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  client_id UUID NOT NULL REFERENCES clients(id),
  session_date DATE NOT NULL,
  raw_notes TEXT NOT NULL,
  created_by UUID NOT NULL DEFAULT auth.uid(),
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  deleted_at TIMESTAMPTZ
);

-- Index for common query patterns
CREATE INDEX sessions_client_id_idx ON sessions (client_id) WHERE deleted_at IS NULL;
CREATE INDEX sessions_session_date_idx ON sessions (session_date DESC) WHERE deleted_at IS NULL;

-- Enable RLS
ALTER TABLE sessions ENABLE ROW LEVEL SECURITY;

-- Authenticated users can read non-deleted sessions
CREATE POLICY "Authenticated users can read sessions"
  ON sessions FOR SELECT
  TO authenticated
  USING (deleted_at IS NULL);

-- Authenticated users can insert sessions
CREATE POLICY "Authenticated users can insert sessions"
  ON sessions FOR INSERT
  TO authenticated
  WITH CHECK (true);

-- Authenticated users can update sessions (for edits and soft delete)
CREATE POLICY "Authenticated users can update sessions"
  ON sessions FOR UPDATE
  TO authenticated
  USING (deleted_at IS NULL)
  WITH CHECK (true);
```

#### `updated_at` trigger

```sql
-- Shared trigger function for auto-updating updated_at
CREATE OR REPLACE FUNCTION update_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER clients_updated_at
  BEFORE UPDATE ON clients
  FOR EACH ROW EXECUTE FUNCTION update_updated_at();

CREATE TRIGGER sessions_updated_at
  BEFORE UPDATE ON sessions
  FOR EACH ROW EXECUTE FUNCTION update_updated_at();
```

### API Endpoints

#### `GET /api/clients?q=<search>`

Returns matching clients for the autocomplete combobox.

- **Query params:** `q` (string, optional) тАФ search term. If empty, returns all clients (capped at 50).
- **Response:** `200 OK` тАФ `{ clients: Array<{ id: string, name: string }> }`
- **Logic:** `SELECT id, name FROM clients WHERE deleted_at IS NULL AND name ILIKE '%' || q || '%' ORDER BY name LIMIT 50`
- **Auth:** Requires authenticated session (enforced by RLS).

#### `POST /api/clients`

Creates a new client. Called by the session save flow when a new client is needed.

- **Request body:** `{ name: string }` тАФ validated with Zod.
- **Response:** `201 Created` тАФ `{ client: { id: string, name: string } }`
- **Error:** `409 Conflict` if a client with the same name (case-insensitive) already exists.
- **Auth:** Requires authenticated session.

## Part 2: Session Capture Form

### Technical Decisions

- **Form library:** `react-hook-form` + `zod` resolver (both already installed). The form schema defines three required fields: `client`, `sessionDate`, and `rawNotes`. The client field uses the `ClientSelection` type from the combobox тАФ the form schema validates that both a name and selection exist.
- **Date picker:** A native HTML `<input type="date">` styled to match the design system. A custom date picker component is unnecessary for this use case тАФ the native input gives us accessibility, keyboard support, and calendar dropdowns for free. Wrapped in a thin `DatePicker` component for consistent styling.
- **Save flow (two-step when new client):** On form submit, the `POST /api/sessions` route handler checks if the client is new (`client_id` is null). If so, it first calls `createNewClient()` to get a real `client_id`, then creates the session. This is a single API call from the frontend тАФ the orchestration happens server-side. If the client creation fails (e.g., race condition duplicate), the session is not created and a 409 is returned.
- **Session service:** A new `lib/services/session-service.ts` handles session creation. It receives the raw form data (including whether the client is new or existing), orchestrates client creation if needed, and inserts the session row.
- **Form reset on success:** After a successful save, the form resets to its initial state: client cleared, date back to today, notes empty. This is handled by `react-hook-form`'s `reset()` method.
- **Toast notifications:** Uses the existing `sonner` toast library (already installed and wired via `components/ui/sonner.tsx` in the root layout).
- **Page layout change:** The capture page stops being a placeholder. It becomes a real page that renders the `SessionCaptureForm` component. The page stays a server component тАФ the form is a client component pushed to `_components/`.

### API Endpoints

#### `POST /api/sessions`

Creates a new feedback session. Optionally creates a new client first if `clientId` is null.

- **Request body:**
  ```json
  {
    "clientId": "uuid-string | null",
    "clientName": "string (required when clientId is null)",
    "sessionDate": "YYYY-MM-DD",
    "rawNotes": "string"
  }
  ```
- **Validation (Zod):** `clientId` is a nullable UUID string. When `clientId` is null, `clientName` must be a non-empty string (max 255). `sessionDate` is a non-empty date string. `rawNotes` is a non-empty string (max 50,000 characters).
- **Response:** `201 Created` тАФ `{ session: { id, client_id, session_date, raw_notes, created_at } }`
- **Errors:**
  - `400` тАФ invalid body / validation failure.
  - `409` тАФ new client name conflicts with an existing client (race condition).
  - `500` тАФ unexpected server error.
- **Auth:** Requires authenticated session (enforced by RLS). The `created_by` column is auto-populated via `auth.uid()` default.

## Part 3: Past Sessions Table

### Technical Decisions

- **Data fetching:** Sessions are fetched server-side via a new `GET /api/sessions` route that supports query params for filtering (`clientId`, `dateFrom`, `dateTo`) and pagination (`offset`, `limit`). Every filter change and "Load more" action makes a fresh server call тАФ no client-side filtering of already-loaded data. This ensures correctness regardless of how many sessions exist.
- **Session list with joined client name:** The `GET /api/sessions` query joins `sessions` with `clients` to return `client_name` alongside each session. This avoids N+1 lookups on the frontend.
- **Client filter combobox:** A new `ClientFilterCombobox` component that reuses the same `Command` + `Popover` pattern as `ClientCombobox` but without the "create new" option. It fetches only clients that have at least one session (via a new `GET /api/clients?hasSession=true` param on the existing route). Includes a "Clear" option to reset the filter.
- **Date range filter:** Two `DatePicker` components (reusing the existing component) for "From" and "To" dates. Both optional, both capped at today. Changing either triggers a fresh fetch.
- **Pagination:** Offset-based. The API returns `{ sessions, total }`. The frontend tracks the offset and compares loaded count to `total` to decide whether to show "Load more". Each "Load more" appends results to the existing list. Changing filters resets the offset to 0 and replaces the list.
- **Expandable rows:** Managed via `expandedSessionId` state. Clicking a row sets it. Only one row expanded at a time. The expanded view renders the same `ClientCombobox`, `DatePicker`, and `Textarea` components used in the capture form, pre-populated with the session's current values.
- **Dirty state tracking for unsaved changes prompt (P3.R9):** The expanded row tracks whether any field has changed from its original value using a simple `isDirty` check (compare current form values to the original session data). When a user clicks another row while `isDirty` is true, a `Dialog` prompt offers Save/Discard/Cancel.
- **Inline edit save flow:** Uses a new `PUT /api/sessions/[id]` route. The request body is the same shape as `POST /api/sessions` (supports changing the client, including to a new client). The service layer handles client creation if needed, then updates the session row.
- **Soft delete:** Uses a new `DELETE /api/sessions/[id]` route that sets `deleted_at = now()` on the session row. The row disappears from the table on success.
- **Refresh mechanism:** The capture form (Part 2) and the past sessions table need to share a refresh trigger. A simple approach: the `CapturePage` holds a `refreshKey` counter in state and passes a `onSessionSaved` callback to the form and a `refreshKey` prop to the table. When the form saves, it calls `onSessionSaved()`, which increments `refreshKey`, which triggers a re-fetch in the table. This requires making the capture page a client component (or using a thin client wrapper).

### API Endpoints

#### `GET /api/sessions?clientId=&dateFrom=&dateTo=&offset=&limit=`

Fetches paginated sessions with optional filters.

- **Query params:**
  - `clientId` (UUID, optional) тАФ filter by client.
  - `dateFrom` (YYYY-MM-DD, optional) тАФ sessions on or after this date.
  - `dateTo` (YYYY-MM-DD, optional) тАФ sessions on or before this date.
  - `offset` (integer, default 0) тАФ pagination offset.
  - `limit` (integer, default 20, max 100) тАФ page size.
- **Response:** `200 OK` тАФ `{ sessions: Array<{ id, client_id, client_name, session_date, raw_notes, created_at }>, total: number }`
- **Logic:** `SELECT sessions.*, clients.name AS client_name FROM sessions JOIN clients ON sessions.client_id = clients.id WHERE sessions.deleted_at IS NULL` plus optional filter clauses. `ORDER BY session_date DESC, created_at DESC`. Returns both the page of results and a total count for the "Load more" decision.
- **Auth:** Requires authenticated session (RLS).

#### `PUT /api/sessions/[id]`

Updates an existing session. Supports changing client (including to a new client).

- **Request body:**
  ```json
  {
    "clientId": "uuid-string | null",
    "clientName": "string (required when clientId is null)",
    "sessionDate": "YYYY-MM-DD",
    "rawNotes": "string"
  }
  ```
- **Validation:** Same Zod schema as `POST /api/sessions` (shared or duplicated).
- **Response:** `200 OK` тАФ `{ session: { id, client_id, session_date, raw_notes, updated_at } }`
- **Errors:** `400` (validation), `404` (session not found or deleted), `409` (duplicate client name), `500`.
- **Auth:** Requires authenticated session (RLS).

#### `DELETE /api/sessions/[id]`

Soft-deletes a session.

- **Response:** `200 OK` тАФ `{ message: "Session deleted" }`
- **Errors:** `404` (session not found or already deleted), `500`.
- **Logic:** `UPDATE sessions SET deleted_at = now() WHERE id = :id AND deleted_at IS NULL`.
- **Auth:** Requires authenticated session (RLS).

#### Updated: `GET /api/clients?q=&hasSession=true`

Existing route extended with an optional `hasSession` param. When `hasSession=true`, only returns clients that have at least one non-deleted session. Used by the client filter combobox.

- **Logic:** Adds an `EXISTS (SELECT 1 FROM sessions WHERE sessions.client_id = clients.id AND sessions.deleted_at IS NULL)` clause to the query.

### Service Layer

#### `lib/services/session-service.ts` (extended)
